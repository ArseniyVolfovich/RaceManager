## Ответы на контрольные вопросы

### 1. Цели и задачи этапа сопровождения программного обеспечения

Основная цель сопровождения заключается в сохранении работоспособности и функциональной пригодности программного продукта после его передачи в эксплуатацию.

К задачам сопровождения относятся:

- поиск и исправление ошибок в программе и данных;
- адаптация программного продукта к изменениям среды эксплуатации;
- добавление новых функций и совершенствование существующих возможностей;
- анализ запросов пользователей и устранение выявленных проблем;
- обновление проектной, технической и пользовательской документации;
- контроль выпуска, распространения и сохранности версий программы;
- тестирование изменений и проверка совместимости компонентов.

В RaceManager сопровождение включает исправление ошибок регистрации и сохранения данных, развитие функций чемпионатов и событий, улучшение интерфейса, обновление базы данных, тестирование серверного API и публикацию новых версий через GitHub.

### 2. Концепция сопровождения

Концепция сопровождения определяет порядок приёма и обработки сообщений об ошибках и запросов на изменение программного продукта. Для каждого изменения устанавливаются причина, приоритет, исполнитель, необходимые ресурсы, способ реализации, критерии проверки и порядок выпуска новой версии.

Изменения должны вноситься последовательно:

1. Регистрация ошибки или требования.
2. Анализ влияния изменения на существующую систему.
3. Создание отдельной задачи или ветки Git.
4. Изменение исходного кода и документации.
5. Сборка и тестирование программы.
6. Проверка результата и объединение изменений.
7. Создание коммита с понятным описанием.
8. Отправка новой версии в удалённый репозиторий.

### 3. Задачи конфигурационного управления

Конфигурационное управление выполняет следующие задачи:

- идентификация компонентов программного продукта;
- хранение исходного кода, документации и настроек;
- учёт версий и истории изменений;
- управление ветками разработки;
- согласованное объединение изменений;
- предотвращение потери и случайной перезаписи файлов;
- возможность возврата к предыдущему стабильному состоянию;
- контроль состава выпускаемой версии;
- обеспечение связи между требованием, изменением и коммитом;
- подготовка сведений о текущем состоянии проекта.

В проекте RaceManager конфигурационными объектами являются клиентские HTML, CSS и JavaScript-файлы, серверные проекты C#, SQL-сценарий базы данных, автоматические тесты, изображения и проектная документация.

## Выполнение практической части

### 1. Проверка установленной системы Git

Для проверки установки используется команда:

```bash
git --version
```

### 2. Подключение проекта к GitHub

Проект RaceManager уже инициализирован как локальный Git-репозиторий и связан с удалённым репозиторием `origin`:

```bash
git remote -v
```

Результат:

```text
origin  https://github.com/ArseniyVolfovich/RaceManager.git (fetch)
origin  https://github.com/ArseniyVolfovich/RaceManager.git (push)
```

### 3. Проверка состояния рабочей копии

```bash
git status
```

Команда показывает текущую ветку, синхронизацию с GitHub, изменённые и новые файлы. Перед созданием коммита проверяется, что в него не попадут временные файлы сборки, локальные настройки и посторонние изменения.

### 4. Добавление результатов практической работы

```bash
git add .gitignore ИТК30_Git.md
```

Для проверки подготовленного набора файлов используется команда:

```bash
git diff --cached
```

### 5. Создание коммита

```bash
git commit -m "docs: выполнить практическую работу ИТК 30"
```

Коммит фиксирует завершённое состояние работы и содержит краткое описание внесённых изменений.

### 6. Отправка изменений на GitHub

```bash
git push origin main
```

После выполнения команды изменения становятся доступны в удалённом репозитории GitHub.

### 7. Просмотр истории

История локальной ветки просматривается командой:

```bash
git log --oneline --decorate --graph
```

В проекте используется ветка `main`, являющаяся современным аналогом ветки `master`, указанной в задании.

Для проверки удалённого состояния выполняются команды:

```bash
git fetch origin
git status
```

Если локальная и удалённая ветки синхронизированы, Git сообщает, что ветка соответствует `origin/main`.
